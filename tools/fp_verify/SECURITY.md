# Security Policy

## Supported scope

This repository covers the public offline verifier (`fp-verify`), example bundles, and the public verification rules documented here.

It does **not** cover private/commercial runtime components that generate bundles in production.

## How to report a vulnerability

Please do **not** open a public issue for a suspected security problem.

Preferred reporting path:

1. Use GitHub's **Private vulnerability reporting** / **Security Advisory** flow for this repository, if enabled.
2. If private reporting is not available, contact the repository maintainer privately through the repository owner's GitHub profile before disclosing details publicly.

## What to include

Please include, when possible:

- affected version or commit
- a short description of the issue
- reproduction steps
- a minimal proof bundle or failing command
- impact assessment
- any suggested remediation

## Disclosure expectations

- Acknowledge receipt as soon as practical.
- Confirm whether the report is in scope.
- Coordinate a fix before public disclosure when reasonable.

Public verification logic is open by design. Production runtime security controls remain private.
